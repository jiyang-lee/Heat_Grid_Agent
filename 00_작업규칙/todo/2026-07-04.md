# 2026-07-04 시뮬레이션 담당 작업

## 오늘 기준 현황

- 현재 저장소에는 프로젝트 개요, 전략 기획, 원본 PreDist 데이터셋, 최종 모델, 시뮬레이션 입력 계약 초안이 있다.
- 사용자가 지금 들고올 수 있는 산출물은 원본 데이터셋과 완료된 자동감지 에이전트 출력이다.
- 이미지 기준 전체 흐름은 우선순위카드, RAG Output, Priority-RAG 연결 기준, DB/JSON 정리, 시뮬레이션 화면 출력 순서다.

## 우선 작업

1. 대표 사례 JSON 고정
   - 기준 파일: `05_시뮬레이션/examples/simulation_input.example.json`
   - 목표: 화면에 필요한 필드가 빠지지 않는지 확인

2. 자동감지 에이전트 출력 연결
   - 입력 위치: `model_detection`, `priority_card`
   - 목표: 모델 결과를 화면용 JSON으로 변환

3. RAG 담당 산출물 연결
   - 입력 위치: `rag_answer`
   - 목표: 원인 설명, 근거, 점검 포인트 표시

4. 연결/DB 담당 산출물 연결
   - 입력 위치: `connection_check`, `db_mapping`
   - 목표: 우선순위와 RAG 설명이 어떤 현장/설비에 붙는지 표시

5. 화면 MVP 구현
   - 목표: 입력 JSON 1개를 읽어 우선순위, 감지 결과, 근거, 점검 포인트를 한 화면에 출력

## 팀에 요청할 것

- RAG 담당: `rag_answer.summary`, `rag_answer.evidence`, `rag_answer.check_points` 형태로 전달
- 연결 담당: Priority-RAG 연결 성공 여부와 기준을 `connection_check` 형태로 전달
- DB/스키마 담당: 현장명, 설비명, 센서 키를 `db_mapping` 형태로 전달
- 자동감지 담당: `(manufacturer, substation_id)`, 라벨, 위험 점수, 시간 구간을 `model_detection` 형태로 전달

## 주의사항

- PreDist의 `substation_id`는 제조사 그룹까지 함께 봐야 안전하다.
- 원본 파일 기준 수, 전처리 윈도우 기준 수, 화면 예시 JSON 기준 수를 섞어 말하지 않는다.
- 시뮬레이션은 모델이나 RAG를 새로 만드는 역할이 아니라, 받은 결과를 운영자가 이해할 수 있게 연결해 보여주는 역할이다.
