# proto 예측모델 신빙성 및 타당성 정량 보고서

## 개요

이 보고서는 `mlmodel2`에서 새 모델을 승격할지 판단하기 위해, 현재 확인된 `proto` 예측모델과 새 학습 후보의 성능 차이를 정량화한 문서다.

중요한 결론은 다음과 같다.

| 판단 항목 | 정량 판단 |
|---|---:|
| proto가 새 학습 후보보다 낫다는 결론 | 높음, 82/100 |
| risk threshold 보정의 실사용 유의미성 | 낮음~중간, 58/100 |
| leadtime proto 유지의 타당성 | 높음, 84/100 |
| mlmodel2에서 새 모델 승격 여부 | 보류 |
| 적용 성격 | 새 모델 승격 아님, proto 보정 후보 |

즉, **현재 확인 범위에서는 proto 예측모델이 가장 낫다.**
`mlmodel2`에서 새로 학습한 후보를 승격하는 것은 타당하지 않고, 실제 적용은 proto 모델 체인 보정 작업에서 별도로 다뤄야 한다.

## 무엇을 했는지

- 고정 전처리 데이터 기준으로 proto 모델과 새 학습 후보를 비교했다.
- risk threshold `0.44 -> 0.40`으로 보정할 경우 기대되는 성능 변화를 계산했다.
- risk 변화량은 holdout 385건 기준으로 bootstrap 10,000회 신뢰구간까지 확인했다.
- leadtime은 holdout 132건 기준으로 bucket별 precision/recall과 bootstrap 95% 구간을 확인했다.
- 결과적으로 새 학습 후보는 폐기하고, proto 예측모델 유지가 타당하다고 판단했다.

## 왜 이렇게 판단했는지

단순히 한 지표가 조금 오른 것만으로 모델을 승격하면 안 된다. 이번 경우 risk는 threshold를 낮추면 recall/F1이 조금 좋아지지만, false positive도 같이 증가한다. 반면 leadtime은 새 학습 후보가 proto보다 명확히 낮다.

따라서 `mlmodel2`의 결론은 다음과 같다.

```text
새 모델 승격: 부적절
proto 모델 유지: 적절
risk threshold 보정: 적용 후보로만 기록
실제 보정 적용: proto 모델 체인 보정 작업에서 수행
```

## 적용 시 기대 변화

보정 후보를 실제 모델 체인에 적용한다면 기대 변화는 다음 정도로 보는 것이 타당하다.

| 항목 | 기대 변화 |
|---|---:|
| risk recall | 약 +4.30% |
| risk F1 | 약 +2.00% |
| risk precision | 약 +0.69% |
| false negative | 약 -10.26% |
| true positive | 약 +4.30% |
| false positive | 약 +3.08% |
| leadtime | 변화 없음, proto 유지 |

이 수치는 **성능 향상이 아니라 risk 민감도 보정 효과**로 해석해야 한다.
즉 “더 좋은 새 모델”이 아니라 “위험 window를 조금 더 잡는 threshold 조정”이다.

## 최종 판단

| 질문 | 답 |
|---|---|
| proto 예측모델이 현재 제일 낫나 | 예, 현재 확인 범위에서는 그렇다 |
| 새 학습 후보를 승격할 만한가 | 아니다 |
| risk threshold 0.40 적용은 유의미한가 | 작지만 recall 개선 목적이면 가능 |

| mlmodel2에 적용 코드를 넣어야 하나 | 아니다 |

최종 결론:

**mlmodel2에서는 proto 예측모델이 현재 최선이라는 판단만 커밋한다.**
실제 proto 모델 체인 보정은 별도 적용 작업에서 해야 하며, mlmodel2에는 보정 적용 코드나 handoff 변경을 남기지 않는다.

## 검증

- risk/leadtime holdout 행 단위 bootstrap 10,000회
- risk threshold `0.44` vs `0.40` 비교
- proto 유지 vs 새 학습 후보 비교
- local Git 상태 확인: `mlmodel2`는 soji 최신 `origin/mlmodel2`에서 문서 커밋만 추가 예정

## 한계와 주의점

- risk F1 delta의 95% bootstrap 구간은 `-0.0051 ~ +0.0288`로 0을 포함한다.
- risk threshold 0.40은 false negative를 줄이지만 false positive도 늘린다.
- `3-7d` bucket은 holdout 6건뿐이라 다음 데이터에서 흔들릴 수 있다.
- proto original report와 current canonical 평가는 holdout row 수가 다르므로, original report 대비 수치는 보조 근거로만 본다.
