# AGENTS.md

## 작업 원칙

- 답변보다 실제 실행을 우선한다.
- 추측을 사실처럼 두지 말고, 가능한 경우 파일/코드/로그/실행 결과를 먼저 확인한다.
- 기존 구조와 패턴을 우선 따르고, 변경 범위는 최소로 유지한다.
- 별도 요청이 없으면 분석에서 멈추지 말고 수정과 검증까지 진행한다.
- 파일 삭제, 대규모 구조 변경, 데이터 손실 가능 작업, 원격 푸시/배포는 명시적 요청 없이 수행하지 않는다.
- 사용자 변경사항은 임의로 되돌리지 않는다.

## 프로젝트 규칙

- Python 패키지 관리는 `uv`를 사용한다.
- 개발 기준은 Python 3.12와 `pyproject.toml` 중심으로 둔다.
- DB는 `PostgreSQL + TimescaleDB` 조합을 우선 고려한다.
- 시계열/운영 로그는 원본과 가공 데이터를 분리해서 저장한다.
- 백엔드는 `repository / service / tool / agent`로 역할을 분리한다.
- Agent 관련 신규 코드는 repo 루트의 `agent/` 아래에서만 작업한다.
- Agent 관련 Git 작업의 기준 브랜치는 `agent1`로 유지한다.

## 문서 운영

- 진행 중 작업은 `docs/todo/YYYY-MM-DD.md`에 기록한다.
- 설계 초안과 작업 분해는 `docs/plan/`에 둔다.
- 구현 결과와 선택 이유, 수치, 검증 근거는 `docs/report/`에 둔다.
- 반복되는 작업은 `report` 형식으로 남기고, 선택 이유와 정량 지표를 함께 기록한다.

## 보고서 규칙

- 보고서는 한국어로 작성한다.
- DB 선택 이유, 아키텍처 선택 이유, 코드 구조 선택 이유를 반드시 남긴다.
- 측정 가능한 값은 가능한 한 모두 수집한다.
- 보고서에는 최소한 다음을 포함한다.
  - 무엇을 했는지
  - 왜 이렇게 했는지
  - 어떤 수치가 나왔는지
  - 남은 한계와 다음 단계

## 결정 근거 기록 원칙

- 코드, 파일, 구조, 기술 선택에는 반드시 why 근거를 남긴다.
- 근거 없는 선택은 기본적으로 미완성으로 본다.
- why 근거는 가능하면 다음 기준 중 하나로 설명한다.
  - 정확성
  - 위험 감소
  - 구현 비용
  - 유지보수성
  - 확장성
  - 디버깅 용이성
  - 기존 프로젝트 규칙과의 정합성
- 보고서와 문서에는 무엇을 했는지뿐 아니라 왜 그렇게 했는지를 반드시 포함한다.
- 가능하면 정량 근거를 우선하고, 정량화가 어렵다면 트레이드오프를 명시한다.

## 작업 흐름

1. 현재 상태 확인
2. 구현
3. 테스트/검증
4. 보고서 정리

## Git 작업 확인

- Agent 작업 전 현재 브랜치가 `agent1`인지 먼저 확인한다.
- Agent 코드가 `agent/` 밖으로 퍼지지 않았는지 점검한다.
