- API 최상위 필수 필드와 `summary/action_plan/caution` output schema는 유지됐다.
- terminal 상태와 anomaly report artifact 종류는 유지됐다.
- decision 순서 차이는 외부 검색 제거, 진단 worker 추가, durable resume 추가로만 분류됐다.
- 자유 텍스트는 exact 비교 대상에서 제외했다.

## Hardening 결과

- PR 02 변경을 A 검색 제거, B 상태 축소, C 정책 레지스트리, D async 경계로 추적 가능하게 문서화했다.
- 신규 외부 검색 candidate/action/task는 거부하고 기존 DB row는 historical read-only로 유지했다.
- worker A/B/C 측정은 각각 `439→439`, `10561→2561`, `5431→5398(no-call)` token이었다.
- clean DB는 fresh `001~004`, predictor/alert 준비, `002/004` 재적용 후 task·ledger·FK·UNIQUE를 확인했다.
- existing DB는 기존 running run을 준비하고 `004`를 두 번 적용해 task 1개만 backfill되고 기존 run/API columns가 보존됨을 확인했다.

## 잔여 위험

프런트 코드는 계획상 수정하지 않았다. 따라서 기존 검색/근거 승인 UI가 historical external row에 수정 요청을 보내면 backend가 409를 반환한다. UI에서 read-only 상태를 표현하는 작업은 별도 범위다.
